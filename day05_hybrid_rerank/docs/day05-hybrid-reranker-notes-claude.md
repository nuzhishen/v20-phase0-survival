## 0. 先写结论(今日一句话技术判断)

在生产级 RAG 系统里,**Dense Retrieval 不能单独承担全部召回**,因为 TMS 场景存在大量错误码(`OTA_TIMEOUT`)、设备型号(`TBOX-3000`)、Android 版本(`Android 11`)和精确术语。

我的设计选择:
- **Dense** 负责**语义召回**(用户换个说法也能召回);
- **Sparse / BM25** 负责**关键词与错误码召回**(精确 token 命中);
- 用 **RRF** 或 **alpha 加权** 做融合;
- 只对候选 **TopN** 做 **Reranker** 重排。

代价是**延迟和复杂度上升**,收益必须用 **Recall / MRR / 退化率 / 延迟** 四类数据验证,不能靠"感觉更好"。

## 1. BM25 / Sparse Retrieval

### 1.1 完整公式

```
                    n              f(qi, D) · (k1 + 1)
score(D, Q) =  Σ  IDF(qi) · ---------------------------------------
                   i=1        f(qi, D) + k1 · (1 - b + b · |D|/avgdl)
```

其中 BM25 的 IDF 项:

```
              /      N - df(qi) + 0.5     \
IDF(qi) = ln | 1 + --------------------- |
              \        df(qi) + 0.5       /
```

LaTeX 形式:

$$\text{score}(D,Q)=\sum_{i=1}^{n}\text{IDF}(q_i)\cdot\frac{f(q_i,D)\cdot(k_1+1)}{f(q_i,D)+k_1\cdot\left(1-b+b\cdot\frac{|D|}{\text{avgdl}}\right)}$$

$$\text{IDF}(q_i)=\ln\left(1+\frac{N-\text{df}(q_i)+0.5}{\text{df}(q_i)+0.5}\right)$$

符号含义:

| 符号 | 含义 |
|---|---|
| `f(qi, D)` | 词 `qi` 在文档 `D` 中的词频(**TF**) |
| `|D|` | 文档 `D` 的长度(词数) |
| `avgdl` | 语料库平均文档长度 |
| `N` | 文档总数 |
| `df(qi)` | 包含 `qi` 的文档数(document frequency) |
| `k1` | TF 饱和系数,常用 1.2 ~ 2.0 |
| `b` | 长度归一化系数,常用 0.75 |

### 1.2 三个组成部分(把公式拆开理解)

BM25 = **IDF 项 ×(TF 饱和项 / 长度归一化项)** 对每个查询词求和。

**① IDF 项 —— 词有多稀有**
越稀有的词区分度越高,权重越大。BM25 的 IDF 加了 `+0.5` 平滑,并套 `ln(1+·)` 保证非负。
> TMS 直觉:`OTA_TIMEOUT` 这种词在语料里 df 极低 → IDF 极高 → 命中它的文档强烈加分。这正是错误码检索的核心动力来源。

**② TF 饱和项 —— 出现越多越好,但边际递减**
朴素 TF-IDF 里 TF 线性增长(出现 10 次 = 10 倍分),不合理。BM25 用饱和函数:

```
TF · (k1 + 1)
--------------      →  TF 增大时分数趋近上界 (k1 + 1),不会无限涨
  TF + k1
```

**③ 长度归一化项 —— 长文档要被惩罚**
分母里的 `k1 · (1 - b + b·|D|/avgdl)`:文档越长(`|D|/avgdl` 越大),分母越大,单个词贡献被压低。防止长文档仅靠"词多"就赢。

### 1.3 `k1`:TF 饱和速度

`k1` 控制词频饱和的**快慢**:

- `k1` **大** → 饱和慢,TF 的影响持续更久(出现 2 次和 10 次差距明显)。
- `k1` **小** → 饱和快,出现一两次就接近上限(高频词不再额外加分)。
- `k1 = 0` 的极端:TF 项退化为常数,只剩 IDF 起作用(纯"有没有这个词")。

### 1.4 `b`:长度归一化强度

`b` 控制文档长度归一化的**程度**:

- `b = 1` → 完全归一化,长文档被充分惩罚。
- `b = 0` → 不归一化,文档长度不影响打分。
- `b = 0.75` → 经验平衡点(大多数语料的默认)。

### 1.5 在 TMS 场景下,`k1`/`b` 真的重要吗?

**关键判断:错误码检索更依赖 IDF,而不是 TF 饱和或长度归一化。**

原因:错误码 / 型号 / 版本号在文档里**通常只出现一两次**(是标识符,不是正文高频词)。所以:
- `k1` 调优对错误码召回**影响有限**(TF 本来就小,饱和曲线的差异体现不出来)。
- `b` 有一点用:短小的"故障处理条目"不该被长篇 wiki 淹没,适度归一化有帮助。
- **真正决定错误码能否命中的是 IDF**(稀有度)+ **分词能否把 `OTA_TIMEOUT` 当成一个 token 保留下来**。

> 这条是面试加分点:能指出"参数调优不是万能的,先保证分词不切碎专有 token,再谈 k1/b"。

## 2. Dense vs Sparse 六维对比

### 2.1 对比表

| 维度 | Dense(向量检索) | Sparse(BM25 / 关键词) |
|---|---|---|
| **语义理解** | 强:同义、近义、改写都能召回 | 弱:只匹配字面词,换说法就丢 |
| **精确匹配** | 弱:易把 `OTA_TIMEOUT` 和 `OTA_FAILED` 判成几乎一样 | 强:精确命中错误码 / 型号 |
| **错误码 / 专有名词** | 弱:embedding 对低频 / OOV token 泛化差 | 强:直接命中倒排里的 token |
| **成本** | 高:embedding 模型推理 + 向量库存储 | 低:倒排索引,CPU / 内存即可 |
| **延迟** | 中高:embed query + ANN 查询 | 低:倒排查表 |
| **冷启动** | 差:要 embed 全库 + 建 HNSW,常需 GPU | 好:建倒排快,无需 GPU |

### 2.2 每一维为什么(落到 TMS 例子)

**语义理解 —— Dense 赢**
用户问"设备升级卡住不动",知识库写的是"OTA 固件推送超时"。字面零重叠,Dense 靠语义能召回,Sparse 直接漏。

**精确匹配 —— Sparse 赢**
`Android 11` 和 `Android 13` 在向量空间里几乎重合(就差一个数字),Dense 分不清;但这在 TMS 里是**不同的兼容分支**,召错就给错方案。Sparse 精确匹配 token 不会混。

**错误码 / 专有名词 —— Sparse 赢**
`OTA_TIMEOUT` 是低频专有 token,embedding 训练时见得少,表示不稳定(OOV / rare token 问题),Dense 容易召回"语义相近但错误码不同"的条目。Sparse 直接在倒排里命中,稳。

**成本 —— Sparse 赢**
Dense 每条 query 要跑一次 embedding 模型 + 向量库检索;Sparse 只是倒排查表,CPU 就够,单位成本低一个量级。

**延迟 —— Sparse 赢**
Dense 链路:`embed(query)`(模型推理)+ ANN 近邻搜索;Sparse 链路:倒排表查询求和。Sparse 通常更低延迟。

**冷启动 —— Sparse 赢**
新语料上线:Sparse 建倒排索引快、无需 GPU;Dense 要把全库过一遍 embedding + 建 HNSW 图,资源和时间成本都高。

> 结论:两者是**互补**关系,不是替代关系。这正是 Hybrid 存在的理由。

## 3. 三种融合策略

融合要解决的核心矛盾:**Dense 分数(余弦相似度,大致 0–1)和 BM25 分数(无上界,可能几十)尺度完全不同,不能直接相加。**

### 3.1 加权求和(Weighted Sum)—— 需先归一化

**第一步:归一化(把两路分数拉到同一尺度)**

```
min-max:  s_norm = (s - s_min) / (s_max - s_min)          → 映射到 [0, 1]
z-score:  s_norm = (s - μ) / σ                            → 均值0方差1
```

**第二步:凸组合**

```
hybrid_score(d) = α · dense_norm(d) + (1 - α) · sparse_norm(d)
```

$$\text{hybrid}(d)=\alpha\cdot\tilde{s}_{dense}(d)+(1-\alpha)\cdot\tilde{s}_{sparse}(d)$$

- `α → 1`:偏 Dense(重语义);`α → 0`:偏 Sparse(重关键词)。
- **适用场景**:两路分数归一化稳定、量纲可控时。
- **弱点**:对**归一化方法敏感**。min-max 受极值影响大(一个离群高分把其它全压扁);z-score 假设近似正态。归一化选择本身是个**隐藏变量**。

### 3.2 线性插值(Linear Interpolation)

```
score(d) = λ · s1(d) + (1 - λ) · s2(d)
```

**和加权求和的关系:代数同形,语义前提不同。**

| | 加权求和(工程习惯) | 线性插值(统计习惯) |
|---|---|---|
| 输入量 | 通常是**归一化后**的检索分数 | 通常是**本身就可比**的量(两个概率、两个 [0,1] 相似度) |
| 典型出处 | Hybrid Retrieval 工程实现 | 语言模型插值 `P = λP1 + (1-λ)P2`、平滑 |
| 是否强制归一化 | 是(否则尺度打架) | 不一定(前提是输入已可比) |
| 作用对象 | 分数 | 分数 / 概率 / logits |

> 面试诚实点:很多 RAG 实现把这两个当**同义词**,严格区别在于"输入量是否已经可比、要不要先归一化、作用在分数还是概率/logits 上"。别装作它们是两个完全不同的算法。

### 3.3 RRF(Reciprocal Rank Fusion)

```
                    1
RRF(d) =  Σ   ---------------
         m∈M   k + rank_m(d)
```

$$\text{RRF}(d)=\sum_{m\in M}\frac{1}{k+\text{rank}_m(d)}$$

- `rank_m(d)`:文档 `d` 在检索器 `m` 结果中的**名次**(从 1 开始);若 `d` 未进入 `m` 的结果,该项记 0。
- `k`:平滑常数,**默认 60**。
- `M`:检索器集合(这里是 {Dense, Sparse})。

**为什么 RRF 有用(核心洞察)**
RRF **只用排名,不用分数**。所以它**天然绕开了尺度不一致问题** —— 不需要归一化,不管 Dense 是 0.83 还是 BM25 是 27.4,只看"你排第几"。这是它相对加权求和最大的工程优势:**少一个隐藏变量(归一化方法)**。

**`k` 的作用**
`k` 抑制头部排名的过度主导:
- `k` 小 → 第 1 名和第 2 名的分差被放大(头部说了算)。
- `k` 大 → 各名次贡献更平滑(排名靠后的也有发言权)。
- 默认 60 是工业界经验值(原始 RRF 论文的推荐),稳健、无需调。

### 3.4 三者选型对比

| | 加权求和 | 线性插值 | RRF |
|---|---|---|---|
| 依赖分数尺度 | 是(必须归一化) | 是(需可比) | **否(只用排名)** |
| 隐藏变量 | 归一化方法 + α | λ | 基本只有 k(且默认稳) |
| 可解释性 | 中(α 有物理意义) | 中 | 高(排名直观) |
| 调参成本 | 需 Grid Search α | 需调 λ | 几乎免调 |
| 何时选它 | 想精细控制语义/关键词配比 | 输入已是可比概率 | 想快速、稳、少踩坑 |

> 今天的做法:**Weighted(跑 5 组 α)必做,RRF 尽量做**。最终选哪个,用 30 条 Query 的 Recall/MRR/退化率**用数据决定**,不拍脑袋。

## 4. Reranker 机制:Bi-Encoder vs Cross-Encoder

### 4.1 Bi-Encoder(双塔,用于召回)

```
query ──► Enc ──► v_q  ┐
                        ├──►  sim(v_q, v_d)  (余弦 / 点积)
doc   ──► Enc ──► v_d  ┘
```

- query 和 doc **各自独立**编码成向量。
- doc 向量可**离线预计算**、建 HNSW 存库;线上只需 embed query + ANN。
- **快、可扩展到百万级**,但 query 与 doc **没有交互**,精度有天花板。
- 这就是 Day4 的 Dense Retrieval 用的东西。

### 4.2 Cross-Encoder(交叉编码,用于精排)

```
[query ; doc] ──► Transformer(full attention)──► 相关性分数
```

- query 和 doc **拼接后一起过模型**,token 级 full attention 充分交互。
- 精度高(能捕捉细粒度匹配,比如"这个错误码到底对不对得上")。
- 但**每个 (query, doc) 对都要过一次完整模型**,**无法预计算**。

### 4.3 为什么更准但更慢(一句话推导)

- **更准**:Cross-Encoder 里 query 的每个 token 能直接 attend 到 doc 的每个 token → 建模 query–doc 的**交互**;Bi-Encoder 压成两个独立向量后,交互信息已经丢了。
- **更慢**:Bi-Encoder 的 doc 向量能预存,线上零成本;Cross-Encoder 无法预存,每个候选都得实时跑一遍完整前向。

### 4.4 Reranker 在管线里的位置 + 成本

```
Query
  │
  ├─ Dense 召回 ─┐
  │              ├─► Hybrid 融合 ──► TopN 候选 ──► Cross-Encoder 重排 ──► Top3
  └─ Sparse 召回─┘        (快、召回导向)              (慢、精度导向,只作用 TopN)
```

**Reranker 四原则:**
1. **只重排候选**,不直接全库检索。
2. **只对 TopN**(默认 Top20 以内),候选数越大越慢。
3. **保留分数**,输出 `rerank_score` 便于分析。
4. **可降级**:BGE 下载/加载失败,自动走 MockReranker,不阻塞闭环。

**成本公式:**

```
Reranker 延迟 ≈ 候选数 N × 单次 Cross-Encoder 推理耗时
```

所以 `N` 必须控制。TopN 取大 → 精度可能更好但延迟线性上升;取小 → 快但可能把正确文档挡在 N 之外。这是精度/延迟的权衡点。

### 4.5 为什么 Reranker 不能替代检索

两个硬理由:

1. **成本不可行**:对 100 万文档每条 query 都跑 Cross-Encoder = `1e6 × 几十ms`,根本跑不动。它天生只能作用于**少量候选**。
2. **救不回没召回的**:Reranker 只能**重排已经召回的候选**。如果粗召回阶段(Dense+Sparse)就把正确文档漏了,Reranker 再准也**无中生有不出来**。这叫**召回天花板**。

> 一句话:**Reranker 提升的是"排序质量",不是"召回率"。召回不行,先修召回,不是加 Reranker。**

## 5. TMS 场景:为什么必须要 Sparse

TMS 知识库里高频出现四类**标识符型**内容,它们恰好是 Dense 的软肋、Sparse 的强项:

| 类型 | 例子 | 为什么 Dense 不行 / Sparse 行 |
|---|---|---|
| **错误码** | `OTA_TIMEOUT`、`E1005`、`DEVICE_OFFLINE` | 低频专有 token,embedding 泛化差,易召回"码不同但语义近"的条目;Sparse 精确命中 |
| **设备型号** | `TBOX-3000`、`XX-Pro-v2` | 型号只有精确匹配才有意义,语义相似度无意义;Sparse token 命中 |
| **Android 版本** | `Android 11` / `Android 13` | 向量空间里几乎重合,但实际是不同兼容分支,召错给错方案;Sparse 区分 |
| **OTA / 固件版本** | `firmware 2.3.1`、`OTA v5` | 版本号是精确字符串约束;Sparse 精确匹配 |

**结论**:这些场景**精确 token 匹配是刚需**。只用 Dense 会**漏召(换个错误码就召不到)或错召(召到相近但不对的)**。Sparse 在 TMS 里不是可选项,是**必需底座**。

> 分词注意(工程细节,面试可提):中文分词器很容易把 `OTA_TIMEOUT`、`Android 11` 切碎。必须用正则保护错误码 / 版本号,不能让 `OTA_TIMEOUT` 被切成 `OTA` + `TIMEOUT` 甚至丢失。这是 Sparse 在中文 TMS 语料上能不能用的前提。

## 6. 今日边界(为什么不做)

### 6.1 为什么不做 Query Rewrite

Query Rewrite 会**引入一个新变量**:改写质量。今天的实验目标是**隔离"融合"这一个因素**,证明 Dense vs Hybrid 的差异到底来自哪。如果同时开 Query Rewrite,一旦指标变好,你**分不清是融合的功劳还是改写的功劳** —— 对照实验被污染。控制变量,今天只测融合。

### 6.2 为什么不做 GraphRAG

Phase 0 的目标是**打通闭环**(设备异常提问 → RAG 检索 → ReAct 决策 → 工具调用 → 处理建议),不是构建知识图谱。GraphRAG 是**复杂度扩展**,不是通关必需件。现在做它会分散资源、拖慢周六通关。分层推进:先把检索底座打稳。

### 6.3 为什么不做 ReAct 深集成

今天专注**检索层质量本身**。ReAct 的深度接入是**周六通关**的事(P1)。今天先把最优检索配置定下来,明天再接进 `lookup_error_knowledge_via_rag()`。**分层交付,不越界**:Day5 交检索,周六交闭环。

## 7. 风险清单

| 风险 | 是什么 | 缓解 |
|---|---|---|
| **权重过拟合** | 30 条 Query 上调出的"最优 α",可能只是过拟合这个小评估集,换一批 Query 就失效 | 看**多个 α 的稳定性**而非单点最优;明确样本量小,α 是**参考不是定论**;必要时优先选更稳健的 RRF |
| **Hybrid 退化** | 某些 Query 上 Hybrid **反而比单路差**(融合把噪声引进来) | **不许造假**,记录退化 Query + 冲突原因(比如纯错误码 Query 被 Dense 拖偏) |
| **Reranker 延迟** | Cross-Encoder 增加显著延迟,候选数越大越慢 | 控制 TopN(20 以内);用数据看"延迟增加 vs 精度提升"是否划算 |
| **候选召回不足** | 粗召回就漏了正确文档,Reranker 救不回(召回天花板) | 先保证**召回阶段**够宽;Reranker 没提升时先查是不是召回漏了,而不是硬吹 Reranker |

## 8. 必须能回答的面试问题(自测 Q&A)

**Q1:为什么 Dense 不够?**
A:TMS 场景大量错误码、设备型号、Android 版本、精确术语。这些是低频专有 token,embedding 泛化差,Dense 易漏召或错召(把 `Android 11` 和 `Android 13` 当一样)。

**Q2:BM25 为什么还有价值?**
A:精确匹配、短文本、专有名词、错误码场景可靠。倒排精确命中 token,不怕"码相近但不同",而且成本低、冷启动快。

**Q3:Hybrid 权重怎么定?**
A:不拍脑袋。在 30 条 TMS/OTT/养老 Query 上做 Grid Search(α=0.2/0.4/0.5/0.6/0.8),用 Recall/MRR/退化率选。

**Q4:RRF 为什么有用?**
A:基于**排名**融合,不依赖两个检索器分数尺度一致。天然绕开 Dense(0–1)和 BM25(无上界)的尺度冲突,还少一个"归一化方法"的隐藏变量。

**Q5:Reranker 为什么不能替代检索?**
A:① Cross-Encoder 成本高,对全库跑不可行,只能作用 TopN;② 只能重排已召回的,救不回漏召的(召回天花板)。它提升排序质量,不提升召回率。

**Q6:为什么今天不做 Query Rewrite?**
A:会引入"改写质量"这个新变量,污染 Dense vs Hybrid 的对照实验。今天要控制变量、隔离融合这一个因素。

**Q7:为什么今天不做 GraphRAG?**
A:Phase 0 目标是通关闭环,不是扩展知识图谱。GraphRAG 是复杂度扩展,不是通关必需。

**Q8:Hybrid 一定比单路好吗?**
A:不一定。会退化。必须记录退化 Query 和冲突原因,用数据说话,不能默认"融合=更好"。

---

## 9. 收尾:面试可背的一句话判断

> 在生产级 RAG 里,Dense 不能单独承担全部召回,因为 TMS 场景有大量错误码、设备型号、Android 版本和精确术语。我的设计是 Dense 管语义召回、Sparse/BM25 管关键词召回,用 RRF 或 alpha 加权融合,只对候选 TopN 做 Reranker 重排。代价是延迟和复杂度上升,收益必须用 Recall、MRR、退化率和延迟数据验证 —— 我不拍脑袋定权重,我用 30 条 Query 的实验数据证明。

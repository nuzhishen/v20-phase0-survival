## 1. BM25 / Sparse Retrieval(08:30–09:15)

### 1.1 Sparse 是什么

**Sparse Retrieval(稀疏检索)** = 基于**词袋 + 词频统计**打分的检索,代表算法 **BM25(Best Matching 25)**。"稀疏"指文档被表示成一个**维度=词表大小、绝大多数为 0** 的向量(只有出现过的词非零),与 Dense 的低维稠密向量正好相反。

本质:**它不理解语义,只统计"Query 里的词在文档里出现得多不多、这些词稀不稀有、文档长不长"**。恰恰因为"只认词、不认义",它对**错误码、编号、专有名词**极其可靠(见 §5)。

### 1.2 BM25 完整打分公式(必须默写)

给定 Query `Q = {q1, q2, ..., qn}`,文档 `D`,BM25 打分:

```
                    n                        TF(qi, D) · (k1 + 1)
score(Q, D) =  Σ   IDF(qi) · ───────────────────────────────────────────────
                   i=1                TF(qi, D) + k1 · (1 - b + b · |D| / avgdl)

其中:
  IDF(qi) = log( (N - df(qi) + 0.5) / (df(qi) + 0.5) + 1 )

  TF(qi, D) = 词 qi 在文档 D 中的原始词频
  |D|       = 文档 D 的长度(词数)
  avgdl     = 语料库文档平均长度
  N         = 语料库文档总数
  df(qi)    = 包含词 qi 的文档数(document frequency)
  k1, b     = 两个可调超参(见 §1.4),常用 k1∈[1.2, 2.0]、b=0.75
```

### 1.3 逐项拆解(TF / IDF 的直觉)

| 项 | 直觉 | 机理 |
|---|---|---|
| **TF(词频)** | 一个词在文档里出现越多,文档越可能相关 | 但**不能线性增长**:出现 100 次不该是 10 次的 10 倍相关 → 分子分母都含 TF,构成**饱和曲线**(TF 越大,增量收益越小) |
| **IDF(逆文档频率)** | 越稀有的词区分度越高,权重越大 | "的/是/了"几乎每篇都有 → df 大 → IDF 趋近 0(近乎无用);"OTA_TIMEOUT"只在少数文档 → df 小 → IDF 大(强区分) |
| **+0.5 平滑** | 防止 df=0 或 df=N 时 log 爆炸/为负 | 拉普拉斯式平滑,保证数值稳定 |
| **log** | 压缩 IDF 的动态范围 | 稀有度的边际效应递减,log 让"极稀有"和"很稀有"不至于差出几个数量级 |

### 1.4 两个旋钮:k1 和 b(面试必考)

| 参数 | 名字 | 控制什么 | 调大后果 | 极端退化 |
|---|---|---|---|---|
| **k1** | 词频饱和度 | TF 增长到多快"饱和" | k1↑ → 词频影响更接近线性(高频词更吃香) | **k1→0**:TF 项 ≈ (0+1)/(1)=1,**词频完全失效**,退化成"只看词在不在(布尔) × IDF";**k1→∞**:饱和消失,退化成**线性 TF**(原始词频直接计权) |
| **b** | 文档长度归一化强度 | 长文档惩罚多重 | b↑ → 长文档被惩罚越狠(避免长文靠堆词得高分) | **b=0**:分母的长度项 =1,**完全不做长度归一化**(长文档天然占便宜);**b=1**:**完全按 `|D|/avgdl` 归一化**(长度惩罚拉满) |

> **记忆锚点**:**k1 管"词频要不要饱和",b 管"长文档要不要惩罚"。** 默认 `k1=1.5, b=0.75` 是通用平衡点 —— 词频有饱和但不失效,长度归一化有但不极端。

### 1.5 降级链(熔断纪律,不许停工)

TMS 语料含大量中文 + 错误码,分词是风险点。降级顺序:

```
BM25(rank_bm25)  ──卡住──►  TF-IDF  ──再卡住──►  关键词命中计分(最低兜底)
```

**分词红线**:无论哪级,都要用正则**保住错误码 / 版本号**(`E1001`、`OTA_TIMEOUT`、`Android 11`),绝不能被中文分词器当噪声切碎丢掉。这是 §5 的命门。

## 2. Dense vs Sparse 6 维对比(09:15–10:00)

> 核心结论:**两者正交互补,不是替代关系。** Dense 抓"意思像",Sparse 抓"字一样"。

| 维度 | Dense(向量 / HNSW) | Sparse(BM25 / 关键词) | 谁赢 & 为什么 |
|---|---|---|---|
| **① 语义理解** | 强:同义、改写、口语↔术语都能拉近("直播卡顿"≈"播放延迟高") | 弱:换个说法就 0 命中,不懂同义 | **Dense** —— 向量空间语义相近即距离相近 |
| **② 精确匹配** | 弱:把编号/专名"语义模糊化",`E1001` 和 `E1002` 向量几乎一样 | 强:命中即高分,字符级精确 | **Sparse** —— 词袋天然精确到 token |
| **③ 错误码 / 专有编号** | 很弱:数字/型号缺乏语义,embedding 无法区分 | 很强:`OTA_TIMEOUT` 稀有词 IDF 高,一命中就冒尖 | **Sparse**(见 §5,这是今天做 Hybrid 的头号理由) |
| **④ 成本** | 高:要 GPU/模型算 embedding + 建 HNSW 图(常驻内存) | 低:纯统计,倒排索引,CPU 即可 | **Sparse** 便宜 |
| **⑤ 延迟** | 中:ANN 快但要算 query 向量 + 查图 | 低:倒排表查询,毫秒级 | **Sparse** 略快 |
| **⑥ 冷启动** | 差:必须先有 embedding 模型,小语料/新词表现不稳 | 好:统计即用,**无需训练**,新语料立即可检索 | **Sparse** —— 开箱即用 |

> **一句话(攻击点)**:"Dense 六维里只在①语义理解一项碾压,但②③④⑤⑥ Sparse 都不吃亏甚至更好 —— 所以生产不是二选一,是**让 Dense 负责语义、Sparse 负责术语,融合两路**。"

## 3. 三种融合策略(10:00–10:30)

> 融合要解决的**核心矛盾**:Dense 分数(余弦 ∈ [-1,1],常见 0~1)和 BM25 分数(0~30+,无上界)**尺度完全不同**,不能直接相加。三种策略,按"是否依赖分数尺度"分成两派。

### 3.1 加权求和(Weighted)—— 依赖归一化

```
dense_norm  = normalize(dense_score)     # 归一到 [0,1]
sparse_norm = normalize(sparse_score)    # 归一到 [0,1]
hybrid_score = α · dense_norm + (1 - α) · sparse_norm       # α ∈ [0,1]
```

- **前提**:必须先 `normalize()`(Min-Max 或 Z-score),否则 BM25 的大数量级会**吞掉** Dense。
- **α 的含义**:α→1 偏 Dense(语义),α→0 偏 Sparse(术语)。今天跑 **α=0.2/0.4/0.5/0.6/0.8** grid search。
- **适用**:两路分数分布相对稳定、可归一化;想要**可解释、可调**的权重旋钮。
- **失效**:归一化对**异常值敏感**(某个 Query 的 BM25 分数暴高会压扁整个分布);跨 Query 的归一化基准不一致时不稳。

### 3.2 线性插值(Linear Interpolation)—— 依赖归一化

```
hybrid_score = α · dense_norm + (1 - α) · sparse_norm       # 形式同加权
```

- **与加权求和的关系**:数学形式相同,但**语义强调不同** —— 线性插值把 α 看成"在两个**分数/概率分布之间插值**"的插值系数,常用于两路输出已是**同尺度概率**(如都过了 softmax / 都归一到概率)的场景。工程里若两路已同尺度,加权求和与线性插值**退化为同一式**;若尺度不同,则线性插值前也必须归一化。
- **适用**:两路分数**已经是同一尺度的概率/相似度**,直接插值即可;想强调"两个分布的连续过渡"。
- **失效**:两路尺度未对齐还直接插值 → 和加权求和一样会被大尺度那路主导。

> **加权 vs 线性插值 一句话**:**都是 `α·A+(1-α)·B`,区别在语义 —— "加权求和"强调"两个可调权重的打分相加","线性插值"强调"两个同尺度分布之间的连续过渡"。落到代码,尺度对齐后是同一行公式。**

### 3.3 RRF(Reciprocal Rank Fusion)—— 不依赖分数,只看排名

```
                    1
RRF_score(d) = Σ  ─────────────         默认 k = 60
               m   k + rank_m(d)

  m         = 遍历每个检索器(Dense、Sparse)
  rank_m(d) = 文档 d 在检索器 m 的排名(1,2,3...,越靠前越小)
  k         = 平滑常数(默认 60),压低头部排名的过度优势、给中部候选机会
```

- **核心优势**:**只用 rank,不用 score** → 天然**免疫尺度不一致**(不管 Dense 是 0~1 还是 BM25 是 0~30,只看"你排第几")。
- **k=60 的作用**:k 越大,头名和第二名的差距越平缓(`1/61` vs `1/62` 很接近),融合更"民主",避免单一检索器的 Top1 一票否决。
- **适用**:**两路分数尺度差异大、难以稳定归一化**时的首选(本项目 Dense vs BM25 正是此情形);鲁棒、少调参。
- **失效**:丢弃了**分数强度信息**(第 1 名比第 2 名强很多 vs 只强一点,RRF 看不见);当分数本身是可靠置信度时,RRF 反而浪费信息。

### 3.4 三者选型速查

| 策略 | 依赖分数尺度? | 主要旋钮 | 何时用 | 何时别用 |
|---|---|---|---|---|
| 加权求和 | 是(需归一化) | α | 想要可解释、可调权重 | 分数分布不稳、异常值多 |
| 线性插值 | 是(需同尺度) | α | 两路已是同尺度概率 | 尺度未对齐还直接插 |
| **RRF** | **否(只用 rank)** | k(默认 60) | **尺度差异大、要鲁棒** | 分数本身是可靠置信度、需保留强度 |

> **今日执行**:先保证 **Weighted + 5 组 α** 跑通;时间够**必须**补 RRF 对比,因为 Dense/BM25 尺度差异大,RRF 很可能更稳(下午用数据验证,不预设结论)。

## 4. Reranker 机制:Bi-Encoder vs Cross-Encoder(10:30–11:00)

### 4.1 两种结构(核心区别:query 和 doc 何时"相遇")

```
Bi-Encoder(双塔,= Day4 的检索器):
   Query ─►[Encoder]─► q向量 ┐
                             ├─► cosine(q, d)   ← 只在最后算一次相似度
   Doc   ─►[Encoder]─► d向量 ┘   (d向量可离线预计算、可建 HNSW 索引)

Cross-Encoder(交叉,= Reranker):
   [Query ⊕ Doc] ─► [Encoder + 逐层交叉注意力] ─► 相关性分数(标量)
        ↑ 拼成一条一起进模型,query 每个 token 都能看到 doc 每个 token
        ↑ 无法预计算 doc:换个 query,整条要重算
```

| 维度 | Bi-Encoder(双塔) | Cross-Encoder(交叉) |
|---|---|---|
| query/doc 交互 | **晚**:各自独立编码,最后才算相似度 | **早**:拼接后逐层交叉注意力,充分交互 |
| doc 能否预计算 | **能**(离线建索引,检索时只算 query) | **不能**(每个 query-doc 对现算) |
| 精度 | 中(交互浅,信息有损) | **高**(全交互,能捕捉细粒度匹配) |
| 速度/成本 | **快**(ANN,O(log N)) | **慢**(每对一次完整前向,O(N)) |
| 定位 | **召回**(从全库捞候选) | **精排**(只对 TopN 排序) |

### 4.2 为什么 Cross-Encoder 更准但更慢

- **更准**:query 和 doc 从第一层就通过**交叉注意力**互相"看到",能捕捉"`Android 11` 对应 doc 里 `安卓11系统`"这类**细粒度、需上下文对齐**的匹配 —— 双塔把两边压成两个向量再比,这种细节被平均掉了。
- **更慢**:doc 向量**无法离线预计算**(它依赖当前 query),所以 **N 个候选 = N 次完整模型前向**。全库百万文档跑一遍 Cross-Encoder = 灾难。→ **只能对召回回来的 TopN(默认 ≤20)精排。**

### 4.3 Reranker 在链路中的位置与成本

```
全库(百万级)
   │  ① 粗召回:Dense + Sparse,各取 Top-K(便宜,O(log N))
   ▼
候选集(Top-K,如 20~50)
   │  ② 融合:RRF / α加权,合成一路(便宜)
   ▼
TopN(如 20)
   │  ③ 精排:Cross-Encoder Reranker,N 次前向(贵,但 N 小)
   ▼
最终 TopK(如 3)──► 交给下游(周六接 ReAct / LLM)
```

**成本分析**:
- Reranker 成本 ≈ **N × 单次 Cross-Encoder 前向延迟**。N=20、单次 ~几 ms → 几十~上百 ms 增量。
- **所以 Reranker 是"精排"不是"召回"**:它**只重排候选,不做全库检索**;把 N 压在 20 以内成本才可控。
- **降级**:BGE-Reranker 下载/加载失败 → 立即启用 **MockReranker**(按 query overlap / expected_keywords / section 命中做规则重排),**绝不因模型问题阻塞今日闭环**。

> **一句话(攻击点 #5)**:"Reranker 准是因为 query-doc 全交叉注意力,但正因为无法预计算、每对都要完整前向,它是 O(N) 的,**只能当 TopN 精排,替代不了 O(log N) 的召回层**。这是分工,不是竞争。"

## 5. TMS 场景:为什么必须要 Sparse(承接 Day4 §5)

TMS(设备管理)语料的四类内容,恰好是 **Dense 的结构性盲区、Sparse 的主场**:

| 内容类型 | 例子 | Dense 为什么召回弱 | Sparse 为什么可靠 |
|---|---|---|---|
| **错误码** | `E1001`、`OTA_TIMEOUT` | 编号无语义,`E1001` 和 `E1002` 在向量空间几乎重合,还可能被"语义化"模糊掉 | 稀有词 IDF 高,精确命中即冒尖,`E1001` 绝不会串到 `E1002` |
| **OTA 相关** | "OTA 升级失败"、`OTA_TIMEOUT` | "OTA"是缩写术语,口语 Query 未必能拉近 | 关键词精确匹配,命中"OTA"就给分 |
| **Android 版本** | `Android 11`、`Android 13` | 数字版本号缺乏语义梯度,11 和 13 向量难区分 | 精确匹配版本号,不同版本不串 |
| **设备型号** | `TMS-X200`、`BOX-3000` | 型号是专有编号,embedding 基本无区分力 | 型号作为稀有 token,IDF 高,精确召回 |

> **一句话**:"TMS 场景一半是**精确术语检索**(用户直接报错误码/型号),纯 Dense 会把这些编号语义模糊化导致召回不稳 —— 这正是 Day4 §5 语义鸿沟里量化记录下来的失败靶子。**Sparse 不是锦上添花,是补 Dense 的结构性盲区。**"

## 6. 今日边界:三件明确不做的事(守纪律)

| 不做的事 | 为什么不做 | 什么时候做 |
|---|---|---|
| **Query Rewrite(查询改写)** | 会**引入新变量,污染 Dense vs Hybrid 的对比实验** —— 一次只动一个变量,今天要干净地量化"Sparse+融合+Reranker"的增量,改写会让归因失效 | 检索层锁定后的后续迭代 |
| **GraphRAG(图检索)** | Phase 0 目标是**通关闭环**,不是扩展复杂知识图谱;GraphRAG 是复杂度扩张,不解决今天要证明的"混合是否优于单向量" | 超出 Phase 0 范围 |
| **LangGraph / ReAct 深集成** | 今天专注检索层的**可评估、可对比**;深集成会把检索问题和决策问题混在一起,无法归因 | **周六通关测试**才把最佳检索配置接入 Day3 ReAct 的 `lookup_error_knowledge_via_rag()` |

> **一句话(攻击点 #6/#7)**:"这些不是不会做,是**故意不做** —— 今天的实验要证明'混合检索是否真优于单向量',任何额外变量(改写/图/Agent)都会污染这个对比。守边界本身就是工程能力。"

## 7. 深集成风险(4 个必须预判的坑)

| 风险 | 是什么 | 怎么防 / 怎么记录 |
|---|---|---|
| **权重过拟合** | α 在 30 条 Query 上调到最优,可能只是**过拟合这 30 条**,换一批 Query 就退化 | 只用 30 条做**方向性判断**,不当成生产定值;记录不同 α 的稳定性,选**最稳**而非最高的那个 |
| **Hybrid 退化** | 掺入 Sparse 后,某些**纯语义 Query 反而变差**(Sparse 引入噪声词命中) | **不准造假说 Hybrid 一定好**;记录退化 Query 比例(退化率)和冲突原因,至少列 5 条失败样例 |
| **Reranker 延迟** | Cross-Encoder 精排 TopN 增加几十~上百 ms,可能超 SLA | 把 N 压在 20 以内;用平均延迟数据验证是否可接受,不硬吹 |
| **候选召回不足** | Reranker **只能重排捞回来的** —— 粗召回没捞到的正确答案,Reranker 也变不出来 | 若 Reranker 没提升,先查**是不是候选里根本没有正确 chunk**(召回上限问题),而非 Reranker 质量问题 |

> **一句话**:"混合检索和 Reranker 都**不是稳赚** —— 权重会过拟合、Hybrid 会退化、Reranker 受召回上限约束。**所以今天的交付不是'我接了 Reranker',而是'我用 Recall/MRR/退化率/延迟数据证明它到底提升还是退化'。**"

---

## 8. 必须理解的问题(自测,合上笔记口述)

1. **为什么 Dense 不够?** → 对错误码 / 设备型号 / Android 版本 / 精确术语召回弱,向量会把编号语义模糊化。
2. **BM25 为什么还有价值?** → 精确匹配、短文本、专有名词、错误码场景可靠且可解释;稀有词 IDF 高一命中即冒尖。
3. **Hybrid 权重怎么定?** → 30 条 Query grid search(α=0.2/0.4/0.5/0.6/0.8),数据说话,不拍脑袋,选最稳。
4. **RRF 为什么有用?** → 基于排名融合、不依赖分数尺度,免疫 Dense(0~1)与 BM25(0~30)的量纲冲突。
5. **Reranker 为什么不能替代检索?** → Cross-Encoder O(N)、无法预计算、每对都要完整前向,只适合 TopN 精排。
6. **为什么今天不做 Query Rewrite?** → 引入新变量,污染 Dense vs Hybrid 对比实验。
7. **为什么今天不做 GraphRAG?** → Phase 0 目标是通关闭环,不是扩展知识图谱。
8. **k1 和 b 各管什么?** → k1 管词频饱和度(→0 词频失效 / →∞ 线性),b 管长度归一化(=0 不归一 / =1 归一拉满)。
9. **Bi vs Cross-Encoder?** → 双塔独立编码可预计算(召回),交叉拼接全注意力更准但 O(N)(精排)。

---

## 9. 今日必须形成的一句话技术判断(背下来)

> 在生产级 RAG 系统中,**Dense Retrieval 不能单独承担全部召回**,因为 TMS 场景存在大量错误码、设备型号、Android 版本和精确术语。我的设计选择是用 **Dense 负责语义召回、Sparse/BM25 负责关键词召回**,再通过 **RRF 或 α 加权融合**,并**只对候选 TopN 做 Reranker 重排**。代价是**延迟和复杂度上升**,收益**必须用 Recall、MRR、退化率和延迟数据验证** —— 不是拍脑袋说 Hybrid 更好,而是用 30 条 Query 的对比实验证明它到底提升还是退化。

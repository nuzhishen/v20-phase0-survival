## 时段提纲

### 08:30–09:15 · BM25 / Sparse Retrieval

**要掌握**
- BM25 主公式的三个组成:IDF 项、TF 饱和项、文档长度归一化项。
- `k1`(TF 饱和速度)、`b`(长度归一化强度)分别是什么直觉。

**公式推导步骤(自己一步步写出来)**
1. 先写朴素 TF-IDF:`TF × IDF`,指出它的问题(TF 线性增长不饱和、不考虑文档长度)。
2. 引入 TF 饱和:把 `TF` 换成 `TF·(k1+1) / (TF + k1)`,说明为什么高频词要边际递减。
3. 引入长度归一化:分母里加 `k1·(1 - b + b·|D|/avgdl)`,说明长文档为什么要被惩罚。
4. 写出完整 BM25,再单独写 IDF 的 BM25 形式。

**面试追问**
- k1 调大调小分别对什么样的 Query 有影响?(高频词多的长文本 vs 错误码这种低频短词)
- 错误码场景里,`k1` 和 `b` 真的重要吗?为什么错误码更依赖 IDF 而不是 TF 饱和?
- 中文分词把 `OTA_TIMEOUT`、`Android 11` 切碎了会怎样?你怎么兜底?

### 09:15–10:00 · Dense vs Sparse

**要掌握**
- 六维对比:语义理解 / 精确匹配 / 错误码与专有名词 / 成本 / 延迟 / 冷启动。

**推导步骤**

2. 再说 Sparse 的本质:倒排索引 + 词频统计 → 强在"字面命中",弱在"换个说法就召不回"。

### 10:00–10:30 · 三种融合策略

**要掌握**
- 加权求和(需先归一化)、线性插值、RRF 三个公式。
- 核心洞察:**RRF 用排名,不用分数,所以天然不怕 Dense 分数(余弦 0–1)和 BM25 分数(无上界)尺度不一致。**

**推导步骤**
1. 加权求和:先写归一化(min-max / z-score),再写 `α·dense_norm + (1-α)·sparse_norm`,指出它对归一化方法敏感。
2. 线性插值:写出 `λ·s1 + (1-λ)·s2`,并说清它和加权求和"代数同形、语义前提不同"在哪(输入量是否已可比 / 是否作用在概率或 logits 上)。
3. RRF:写出 `Σ 1/(k + rank_m(d))`,解释 `k`(默认 60)抑制头部主导,以及为什么不需要归一化。

**面试追问**
- 为什么不直接把 Dense 分数和 BM25 分数相加?(尺度不一致,BM25 会碾压)
- 归一化用 min-max 还是 z-score,对结果影响大不大?这是不是一个隐藏变量?
- RRF 的 `k` 调大调小是什么效果?为什么工业界默认 60?
- 加权求和 vs RRF,你今天的数据上会怎么选?靠什么证明?(30 条 Query 实验,不拍脑袋)

### 10:30–11:00 · Cross-Encoder vs Bi-Encoder(Reranker)

**要掌握**
- Bi-Encoder:双塔独立编码,可预计算,快;Cross-Encoder:拼接后 full attention,精度高,不可预计算,慢。
- Reranker 在管线里的位置(融合之后,只对 TopN)+ 成本 = 候选数 × 单次推理。

**推导步骤**
1. 画两条数据流:Bi-Encoder `Enc(q)` 和 `Enc(d)` 分开算 → 相似度;Cross-Encoder `CE([q; d])` 一起过模型 → 直接出分。
2. 从"有没有 query–doc token 级交互"推出精度差异。
3. 从"能不能离线预计算 doc"推出速度/可扩展性差异。
4. 得出架构结论:粗召回(快)→ 精排(慢,只作用 TopN)。

**面试追问**
- 为什么不对全库每个文档跑 Cross-Encoder?(100 万文档 × 单次几十 ms = 不可行)
- Reranker 能不能救回粗召回阶段漏掉的正确文档?(不能 —— 召回天花板)
- TopN 取 20 还是 100,你怎么权衡精度和延迟?
- Reranker 上了但 Top1/MRR 没提升,你会怎么归因?(候选召回不足 / Mock 质量不够,不硬吹)

## 面试追问清单(汇总·可直接自测)

1. 为什么 Dense 不够?→ 错误码 / 设备型号 / Android 版本 / 精确术语召回弱。
2. BM25 还有价值吗?→ 精确匹配、短文本、专有名词、错误码可靠。
3. Hybrid 权重怎么定?→ 30 条 Query 做 Grid Search,不拍脑袋。
4. RRF 为什么有用?→ 基于排名融合,不依赖不同检索器分数尺度一致。
5. Reranker 为什么不能替代检索?→ Cross-Encoder 成本高,只适合 TopN 精排,且救不回没召回的。
6. 为什么今天不做 Query Rewrite?→ 会引入新变量,污染 Dense vs Hybrid 对照实验。
7. 为什么今天不做 GraphRAG?→ Phase 0 目标是通关闭环,不是扩展知识图谱。
8. Hybrid 一定比单路好吗?→ 不一定,会退化;必须记录退化 Query 和冲突原因。
9. 30 条上的最优 alpha 可信吗?→ 样本小,警惕过拟合,看稳定性不看单点。

## 今日必须形成的一句话判断(留给正式笔记展开)

> 在生产级 RAG 里,Dense 不能单独承担全部召回,因为 TMS 场景有大量错误码、设备型号、Android 版本和精确术语。设计选择是 Dense 管语义召回、Sparse/BM25 管关键词召回,用 RRF 或 alpha 加权融合,只对候选 TopN 做 Reranker。代价是延迟和复杂度上升,收益必须用 Recall / MRR / 退化率 / 延迟数据验证。

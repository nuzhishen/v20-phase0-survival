## 1. RAG 六阶段全链路（08:00–09:00）

### 1.1 数据流总图（手写六阶段）

RAG = **Retrieval-Augmented Generation**，本质是"用外部知识检索结果增强 LLM 的生成上下文"。把它拆成离线（Indexing）与在线（Query-time）两条流水线：

```
═══════════════ 离线 Indexing（建库，慢，可重跑）═══════════════
 ① Load          ② Chunk          ③ Embed          ④ Store
 原始文档     →   切片            →  向量化         →  写入向量库
 (md/pdf/db)      (Chunker)         (Embedding)       (Qdrant)
   │                │                  │                 │
 解析/清洗       策略化分块         dense vector       HNSW 图 + Payload
 去噪/规整       带 metadata        [0.13, -0.7, ...]  domain 索引

═══════════════ 在线 Query-time（查询，快，每次请求）═══════════
 ⑤ Retrieve                              ⑥ Evaluate / Generate
 query → embed → ANN 搜索 → top_k       →  评估检索质量（今天）
                  │                          ／ 喂给 LLM 生成（Day5+）
            domain 过滤 + score_threshold
```

> **Day4 覆盖范围**：① Load → ② Chunk → ③ Embed → ④ Store → ⑤ Retrieve → **⑥ 仅做 Evaluate，不做 Generate**。
> **为什么砍掉 Generate**：检索是确定性系统（同 query 同库 → 同结果），生成是概率系统。先把确定性的部分量化到位（Recall/MRR/延迟），再叠加 LLM 的不确定性，否则出了幻觉无法定位是"没检索到"还是"检索到了但 LLM 不会用"。

### 1.2 六阶段逐阶段拆解（每阶段的工程关注点）

| 阶段 | 输入 | 输出 | 工程关注点（面试会问） | Day4 是否手写 |
|---|---|---|---|---|
| ① Load | 原始文档 | 清洗后的纯文本 | 编码、去页眉页脚、表格/代码块保留 | 语料手写 |
| ② Chunk | 纯文本 | `List[Chunk]` | chunk_size / overlap / 边界对齐 / metadata 注入 | **核心手写** |
| ③ Embed | chunk.text | `vector[d]` | 模型选型、归一化、batch、确定性（Mock 用 hash） | 骨架手写 |
| ④ Store | vector + payload | Qdrant Point | HNSW 参数、距离度量、Payload 索引 | **HNSW 手写** |
| ⑤ Retrieve | query | top_k results | query 向量化、ANN、过滤、阈值、latency | Pipeline 手写 |
| ⑥ Evaluate | results vs GT | 指标 | Recall@K/MRR/低分率/Context Precision | **必须手写** |

### 1.3 Chunk 策略对比（这是 Day4 的灵魂，三域差异化选型）

切片是整个 RAG 里"信息论上信息损失最大"的一步——切错了，后面再好的 Embedding 和 Reranker 都救不回来。

| 策略 | 原理 | 优点 | 缺点 | 我的选型场景 |
|---|---|---|---|---|
| **固定长度 Fixed** | 按字符/token 数硬切 + overlap | 简单、可控、延迟稳定 | 切断语义、破坏句子 | **养老**（结构松散，用固定+边界回退） |
| **语义分块 Semantic** | 按句向量相似度断点切 | 语义完整 | 需额外算句向量，慢，建库成本高 | 暂不用（Day4 红线，建库太慢） |
| **递归分块 Recursive** | 按层级分隔符（`#`→`段落`→`句`）递归切 | 尊重文档结构，适配 md | 实现略复杂 | **TMS**（按异常码段落 `##` 切，chunk≈1024） |
| **QA 对分块** | 一个 Q+A 为一个原子单元 | 检索粒度=回答粒度，召回精准 | 仅适合 FAQ 结构 | **OTT**（FAQ 天然 Q/A，chunk≈512） |

**三域选型理由（手写业务判断，CTO 会追问"为什么不一刀切"）**：

- **TMS（递归，1024）**：故障手册以"异常码 + 现象 + 排查 + 建议"为段落单元。一个 chunk 必须装下"一个完整故障的诊断闭环"，否则 Agent 检到"现象"却检不到"处置建议"，等于没检。`##E1001` 这种 markdown 标题是天然递归边界。
- **OTT（QA 对，512）**：运维 FAQ 本身就是问答结构。**检索粒度应当等于回答粒度**——用户问一个问题，理想情况召回的就是对应那一条 Q/A，不掺别的。所以一个 Q+A = 一个 chunk，512 足够。
- **养老（固定 + 边界，512）**：健康知识结构不规整（有的按疾病、有的按症状），没有稳定的层级分隔符。用固定长度保证延迟可控，但加"句子边界回退"避免把血压数值 `120/80` 从中间切断。**overlap 设 50–80**，防止跨 chunk 的禁忌信息丢失。

> **一句话提炼**：切片策略 = 文档结构 × 检索粒度 × 业务最小语义单元。没有银弹，所以我写了三个 Chunker 而不是调一个库。

## 2. 向量检索内核（09:00–10:00）

### 2.1 Embedding 空间的本质

- Embedding 把文本映射到 $\mathbb{R}^d$ 的稠密向量（BGE-base 是 768 维），**语义相近 → 向量空间距离近**。
- 相似度度量三选一，RAG 默认 **Cosine（余弦）**：

$$
\text{cos}(q, d) = \frac{q \cdot d}{\|q\|\,\|d\|}
$$

  - **为什么用 Cosine 不用 L2 欧氏**：文本向量更关心"方向（语义朝向）"而非"模长（绝对幅度）"。归一化后 Cosine 与点积（Dot）等价，Qdrant 里我用 `Distance.COSINE`。
  - **归一化前提**：BGE 类模型输出建议先 L2 归一化，否则模长会污染相似度。Mock Embedding 也要保证确定性（同 text → 同 vector，用 hash 种子）。

### 2.2 ANN vs KNN：为什么不能暴力检索

| | KNN（暴力） | ANN（近似，HNSW） |
|---|---|---|
| 做法 | query 与库里**每个**向量算距离，全排序 | 在图上贪心跳转，只算极小部分 |
| 复杂度 | **O(N·d)** 每查询 | **≈ O(log N · d)** |
| 召回 | 100%（精确） | 95–99%（可调，牺牲 1–5% 换百倍速度） |
| 适用 | N < 万级 | N 百万级（我的场景：1.2M 设备级知识必然走 ANN） |

> **关键认知**：ANN 是用"召回率"换"延迟"的工程妥协。生产级一定走 ANN，但要**有意识地知道自己丢了几个 %**，这就是 `ef_search` 旋钮的意义。

### 2.3 HNSW 原理（DeepSeek 深度 · 手写分层图）

**HNSW = Hierarchical Navigable Small World**，多层可导航小世界图。

```
        ┌──────────────────────────────────────────────┐
 Layer2 │   ●───────────────────────────●              │  稀疏，长跳（高速公路）
        │   │                           │              │
 Layer1 │   ●──────●──────────●─────────●──────●        │  中等密度
        │   │      │          │         │      │        │
 Layer0 │   ●─●─●─●─●─●─●─●─●─●─●─●─●─●─●─●─●─●─●─●─●     │  全量节点，最密（市区小路）
        └──────────────────────────────────────────────┘
          搜索：从顶层 entry point 贪心找最近 → 下沉一层 → 重复 → Layer0 精搜
```

**为什么快**：上层稀疏图负责"粗定位、长距离跳转"（coarse），逐层下沉到底层稠密图做"细搜索"（fine）。每层贪心走到局部最优再下沉，整体 ≈ O(log N)，而不是 O(N) 扫全库。本质和"跳表（Skip List）"是同一个思想用在了图上。

**三个核心参数的 trade-off（必须背熟，面试高频）**：

| 参数 | 含义 | 调大的影响 | 我的线上值 | 理由 |
|---|---|---|---|---|
| **m** | 每个节点的最大连接数（图的稠密度） | 召回↑、内存↑、构建慢 | **16** | 16 是召回/内存的经典平衡点，>32 收益递减 |
| **ef_construct** | 建库时的搜索深度（图质量） | 图质量↑、召回↑、**建库慢** | **100** | 一次性成本，建库可慢，质量优先 |
| **ef_search** | 查询时的候选队列深度 | 召回↑、**查询延迟↑** | **64** | 在线旋钮，平衡召回与 P99 延迟 |

> **记忆口诀**：`m` 定图的骨架（结构，定死后改不了要重建），`ef_construct` 定建库质量（离线，可慢），`ef_search` 是唯一的"在线召回/延迟旋钮"（可热调）。
> **生产经验值**：`m=16, ef_construct=100, ef_search=64` 是中等规模（百万级）的安全起点。召回不够先调大 `ef_search`（不用重建库），还不够再重建并调大 `m`。

### 2.4 Qdrant 数据模型（手写配置参数）

```
Collection（= 一张向量表）
   │
   ├─ vectors_config: {size=768, distance=COSINE}   ← 向量维度 + 度量
   ├─ hnsw_config:    {m=16, ef_construct=100, ef_search=64}  ← 手写
   │
   └─ Point（= 一行）
        ├─ id        : 唯一标识（chunk_id）
        ├─ vector    : [0.13, -0.7, ...] (768 维)
        └─ payload   : {domain, text, metadata...}  ← 结构化过滤字段
              │
              └─ domain 字段建 KEYWORD 索引 ← 多租户/多领域过滤加速
```

**为什么给 `domain` 单独建 Payload 索引**：TMS/OTT/养老三域的知识混在一个 Collection 里，检索时必须能"先按 domain 硬过滤再做向量搜索"，否则 OTT 的"卡顿"会污染 TMS 的检索结果。Payload 索引把过滤从 O(N) 扫描降到索引查找，这是生产级多租户隔离的标准做法。

## 3. RAG 评估与失效模式（10:00–11:00）

### 3.1 语义鸿沟（Semantic Gap）——今天必须刻进 DNA 的概念

> **定义**：用户的口语化 Query 与文档的书面语表述，在向量空间里映射到了**不够近的位置**，导致 Dense 检索召回失败。

**铁案例（必须背，明天混合检索的唯一理由）**：

```
用户 Query: "直播卡顿"            （口语、短、用户视角）
文档原文:   "OTT 播放延迟高"      （书面、术语、系统视角）
            "码率自适应失败"
            "CDN 节点回源超时"

向量相似度：可能只有 0.4~0.5，跌破 score_threshold → 召回失败 ❌
```

**根因分析（三层）**：
1. **词面不重叠**："卡顿" vs "延迟/超时"，纯关键词匹配（BM25）这里反而可能命中"卡顿"如果文档有这词，但 Dense 又可能因为语义被术语稀释而错过。
2. **领域术语 gap**：用户不懂"码率自适应""回源"，但这才是文档的索引词。
3. **粒度 gap**："直播卡顿"是症状，文档是"排查步骤"，问题和答案在向量空间不天然靠近。

**解法演进路线**：
- **Day4（今天）**：纯 Dense，先承认这个鸿沟存在，把失败案例**记录到 `day04-what-blocked.md` 第 2 问**，作为 Day5 的弹药。
- **Day5**：**Hybrid = Dense + Sparse(BM25)**。BM25 负责"词面精确命中"（catch "卡顿" 这种关键词），Dense 负责"语义泛化"，两者用 RRF（Reciprocal Rank Fusion）融合，召回率从 ~60% → ~85%。
- **更后续**：Query Rewrite / HyDE（让 LLM 先把口语改写成术语再检索）——V20 红线内禁止，先不碰。

### 3.2 评估指标体系（RAGAS 四件套 + 工程八指标）

**先分清两层职责（CTO 必问的陷阱题）**：

> **"RAG 检索到了，但 LLM 还是幻觉，是谁的锅？"**
> 答：拆成两个独立问题分开评估——
> - **RAG 的职责**："库里有没有相关知识 + 有没有把它检索出来" → 用 **Recall / Context Precision** 量化。
> - **LLM 的职责**："检索出来的对的知识，会不会被正确使用、不编造" → 用 **Faithfulness / Answer Relevancy** 量化。
> 今天只测 RAG 侧（检索侧），所以**不接真实 LLM 生成**。

**RAGAS 四指标概念（Day4 只理解，今天只实测检索侧两个）**：

| 指标 | 衡量什么 | 归属层 | Day4 是否实测 |
|---|---|---|---|
| **Context Precision** | 召回的 top_k 里，相关 chunk 的占比（精不精） | 检索侧 | ✅ 人工标注 |
| **Context Recall** | 该召回的相关知识，召回全了没（全不全） | 检索侧 | ✅ 用 Recall@K 近似 |
| **Faithfulness** | LLM 答案是否忠于检索到的上下文（编没编） | 生成侧 | ❌ 明天起 |
| **Answer Relevancy** | LLM 答案与用户问题的相关度（答没答到点） | 生成侧 | ❌ 明天起 |

**今天实跑的工程八指标（基线表）**：

| 指标 | 计算方式 | Day4 基线门槛 | W8 目标 |
|---|---|---|---|
| Recall@1 | Top1 命中数 / 总 query 数 | ≥ 40% | ≥ 60% |
| Recall@3 | Top3 命中数 / 总数 | ≥ 60% | ≥ 75% |
| Recall@5 | Top5 命中数 / 总数 | ≥ 50% | ≥ 80% |
| MRR | 平均 (1 / 首个命中的排名) | 记录 | ≥ 0.6 |
| Context Precision | topK 中相关 chunk 占比（人工标注） | ≥ 60% | ≥ 75% |
| No-answer / Fallback 正确率 | 无关 query 正确返回空/低置信 | ≥ 80% | 100% |
| 低分率 | score < 0.5 的 query 比例 | < 20% | < 10% |
| 平均检索延迟 | 30 条 query 平均耗时 | 记录 | < 200ms |

**MRR 直觉**：如果命中排在第 1 位 → 贡献 1.0；第 2 位 → 0.5；第 3 位 → 0.33。它惩罚"虽然召回了但排得靠后"，比单纯 Recall 更能反映"用户/LLM 实际会先看到的东西对不对"。

**为什么要测 No-answer / Fallback**：生产级 RAG 最危险的不是"没召回"，而是"对一个根本无关的 query（如'今天天气'）强行召回一堆低分 chunk 塞给 LLM，诱发幻觉"。所以 `score_threshold` 必须存在——低于阈值就返回空/低置信，宁可不答也不乱答。这是 **No-answer 能力**，是可靠性工程在 RAG 里的体现。

### 3.3 三大失效模式速查（面试可直接背）

| 失效模式 | 现象 | 根因 | 解法 |
|---|---|---|---|
| **语义鸿沟** | 该召回的没召回 | 口语↔术语向量映射失效 | Hybrid（Dense+BM25） |
| **切片污染** | 召回的 chunk 不完整/掺杂 | chunk 太碎（断语义）或太大（掺无关） | 调 chunk 策略 + overlap |
| **低分撑爆** | 无关 query 也召回一堆 | 缺 score_threshold | 设阈值 + No-answer 兜底 |

## 4. 面试攻击点防御（贯穿全程，逐条拆解）

> 这一节是把上面的原理**翻译成 CTO 追问下的口头答案**。每条都要能 30 秒脱稿。

| # | 追问 | 我的应答要点 |
|---|---|---|
| 1 | "Chunk size 怎么选？" | 不是拍脑袋调参，是按文档结构定。TMS 按异常码段落递归切（≈1024，保证诊断闭环），OTT 按 QA 对（≈512，检索粒度=回答粒度），养老按固定+句子边界（≈512，overlap 50–80 防禁忌信息丢失）。**核心：chunk 的边界应对齐业务最小语义单元**。 |
| 2 | "Qdrant vs Chroma / Pinecone / ES？" | Qdrant：可自部署（数据合规可控）、原生 HNSW、Payload 过滤强（多租户隔离刚需）。Pinecone 是托管 SaaS 不可自部署；Chroma 偏原型；ES 是稀疏检索为主、向量是后加的。我的场景要按 domain/区域/版本做结构化过滤 + 百万级向量，Qdrant 最契合。 |
| 3 | "HNSW 的 m 和 ef_construct 怎么调？" | `m`=图骨架稠密度，↑则召回↑但内存↑、建库慢，定死要重建，线上取 16。`ef_construct`=建库搜索深度，↑则图质量↑但建库慢，离线成本可接受，取 100。在线只调 `ef_search`（64）平衡延迟与召回，不用重建。 |
| 4 | "语义鸿沟怎么解决？" | 今天先承认并记录失败案例（"直播卡顿"↔"播放延迟高"）。明天上 Hybrid：BM25 抓词面、Dense 抓语义，RRF 融合，召回 60%→85%。再激进可上 Query Rewrite/HyDE，但 V20 红线内先不碰。 |
| 5 | "RAG 检索到了但 LLM 还是幻觉，谁的问题？" | 分层评估。RAG 负责"有没有相关知识 + 检没检出来"（Recall/Context Precision）；LLM 负责"会不会用、编不编"（Faithfulness/Answer Relevancy）。两层独立打分，先排查是 Context Precision 低（检索锅）还是 Faithfulness 低（生成锅）。 |

## 5. 今日必须形成的一句话技术判断（脱稿基线）

> 在生产级 RAG 系统中，**纯 Dense Retrieval 不能解决语义鸿沟问题**，因为用户口语化 Query（"直播卡顿"）与文档书面语（"OTT 播放延迟高"）的向量空间映射可能失效。我的设计选择是 **Day4 先建立 Dense 基线、Day5 引入 BM25 稀疏向量做混合检索、用 Payload 过滤确保领域隔离**，代价是增加了索引维护复杂度，收益是召回率从纯向量的 ~60% 提升到混合检索的 ~85%。

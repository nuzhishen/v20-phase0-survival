# Day 2 面试攻击点

## 1. 截断 JSON 如何在架构层杜绝？

我不会在业务层直接解析 LLM 文本，而是在统一响应层先检查 `finish_reason`。`length` 立即抛出 `TruncationError`，残缺内容不得进入 JSON 解析和 Tool Calling。恢复路径是压缩历史、减少工具 Schema、拆分任务或降级到规则引擎。`tool_calls` 则进入独立分支，对工具名、参数 Schema、权限、风险级别和幂等键逐项校验。这样把非确定性响应隔离在工具执行之前。

## 2. 日预算只剩 1%，紧急 OTA 是否拦截？

默认拦截 LLM 路径，因为紧急不等于可以突破成本和安全边界。业务兜底分三层：优先使用已经审核的规则模板生成升级计划；确需大模型时发起带预算上限的人工审批；审批失败则只输出设备清单和风险报告，不自动下发 OTA。预算断路器保护的是不可逆成本，高危操作仍必须经过 HITL。

## 3. DeepSeek 返回 503 如何退避？

503 属于可重试的服务端瞬时故障。我会采用有限次数的退避，例如 1 秒、2 秒并加随机抖动，最多两次。持续失败后立即停止重试，触发 Provider 熔断，路由到备用模型或规则引擎，并记录 request ID、重试次数和最终降级原因。无限重试会放大故障并消耗预算。

## 4. 为什么 finish_reason=length 不直接重试？

相同上下文、相同输出上限的原样重试不会改变截断条件，只会再次付费。正确做法是改变约束：压缩 History、裁剪 Tools Schema、降低任务输出规模或拆分请求。它是恢复任务，不是对同一请求的 Retry。

## 5. Pre-flight 估算不准确怎么办？

Pre-flight 不是精确账单，而是最坏情况保护。请求前按估算输入加最大输出预留预算；响应后按 Provider 返回的真实 usage 结算并释放差额。估算器需要按模型 tokenizer 校准，并保留安全余量。没有 Pre-flight 时，任何误差都会在费用发生后才被发现。

## 30 秒话术

我把 LLM 调用看成受预算和协议约束的外部资源。请求前按输入、历史、工具 Schema 和最大输出做 Pre-flight，超限直接降级；响应后先按 `finish_reason` 状态机分流，`length` 禁止解析和原样重试，`tool_calls` 必须经过参数、权限与幂等校验；网络类错误才进入有限重试，重试耗尽后熔断并切备用模型或规则引擎。这样同时控制成本、截断风险和故障放大。

## 今日技术判断

在生产级 Agent 系统中，`finish_reason` 必须作为 Runtime 状态机输入。我的设计是在统一响应层区分 `stop`、`tool_calls`、`length`、`content_filter` 和未知状态；其中 `length` 立即抛出 `TruncationError`，禁止解析残缺 JSON，也禁止原样重试，转入上下文压缩、规则 Fallback 或 HITL。代价是部分长文本任务不能自动完成，收益是阻断截断内容进入高风险工具链。

# Temperature / Top_p 参数路由

## 1. 原理

`temperature` 调整概率分布的尖锐程度：

- 越低越倾向高概率 token，输出更稳定。
- 越高越增加低概率 token 被采样的机会。

`top_p` 使用 nucleus sampling，只从累计概率达到阈值的候选集合中采样。

工程原则：通常优先调一个参数，避免同时大幅调整后无法解释效果。即使参数为 0，也不代表绝对确定性，Provider、模型版本和推理后端仍可能带来差异。

## 2. 决策矩阵

| 场景 | Temperature | Top_p | 理由 |
|---|---:|---:|---|
| TMS 设备故障诊断 | 0.0-0.2 | 0.1 | 输出稳定，降低幻觉 |
| 养老用药咨询 | 0.0 | 0.1 | 高风险场景，禁止发散 |
| OTT 运营报告生成 | 0.5-0.7 | 0.9 | 允许有限的结构化表达 |

## 3. 生产级补充

参数路由不能代替安全机制：

- TMS 诊断仍需 Schema 校验、知识证据和 HITL。
- 养老用药不能因为 temperature=0 就允许剂量建议。
- OTT 报告可以发散，但事实字段必须来自结构化数据。

推荐把参数路由配置化：

```text
task_type -> model -> temperature -> top_p -> max_completion_tokens
```

这样才能审计、回滚和做回归测试。

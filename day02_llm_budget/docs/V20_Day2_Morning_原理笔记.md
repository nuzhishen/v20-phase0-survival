## 一、Token经济学：成本感知是生产级Agent的第一道墙

### 1.1 你必须先建立的基本认知

Token不是字符，不是单词。不同模型的Tokenizer不同：

| 语言 | 平均Token换算 | 说明 |
|------|--------------|------|
| 英文 | 1 Token ≈ 0.75 个单词 | 大部分英文单词是1个Token |
| 中文 | 1 Token ≈ 0.5-1 个汉字 | 汉字通常1-2 Token，取决于Tokenizer |
| JSON键名 | 1-3 Token | 如 `"device_id"` ≈ 3 Tokens |
| 代码/符号 | 1 Token/符号 | `{`, `}`, `"` 通常各1 Token |

**关键认知：你以为写了200字，实际LLM API账单上是450 Tokens。**

### 1.3 Pre-flight Token预算检查（为什么必须在请求前做？）

**反面教材理解：**
```
❌ 错误做法（事后统计）：
   1. 发送LLM API请求
   2. 等待响应（已经计费）
   3. 从响应头读取 usage.total_tokens
   4. 发现超预算 → 扣费已发生，无法挽回

✅ 正确做法（Pre-flight检查）：
   1. 在发送请求前，用tiktoken/本地计数器估算Token数
   2. 比对预算（全局日预算 / 单任务预算 / 用户配额）
   3. 超预算 → 触发降级（压缩History / 换小模型 / 直接拒绝）
   4. 通过 → 发送请求
   5. 记录实际消耗，更新预算余额
```

**三层预算架构（面试必讲）：**
```
Layer 1: 全局日预算         = $50/day
    ↓ 超限 → 拒绝所有非紧急请求
Layer 2: 单任务预算         = $0.5/task（如单次OTA升级计划）
    ↓ 超限 → 降级到小模型 or 触发Fallback规则引擎
Layer 3: 用户/区域配额      = $5/user/day
    ↓ 超限 → 该用户请求排队或降速
```

**核心原则**：任何一层失守，都有下一层兜底。成本防线不是单点，是纵深防御。

### 1.4 Token定价的工程影响（以主流模型为参考）

> 注意：具体价格以官方最新公告为准，此处为建立量级认知。

| 场景 | 输入Token量 | 输出Token量 | 工程影响 |
|------|------------|------------|---------|
| 简单设备状态查询 | ~1000 | ~200 | 适合小模型处理 |
| OTA升级计划生成 | ~2000 | ~500 | 需要大模型 |

### 2.2 Lost in the Middle：位置效应是Agent的隐形杀手

**研究结论**（Stanford 2023, "Lost in the Middle"论文）：

```
┌─────────────────────────────────────────────────────────────────────┐
│            LLM对Context不同位置的信息利用率分布                       │
│                                                                     │
│  信息                                                               │
│  利用率                                                             │
│   高 ┤                                                             │
│      │  ████                                          ████         │
│      │  ████                                          ████         │
│   中 ┤  ████                                          ████         │
│      │  ████     ██                              ██   ████         │
│      │  ████     ██       ▄▄               ▄▄    ██   ████         │
│   低 ┤  ████     ████   ████             ████   ████  ████         │
│      └──────────────────────────────────────────────────────────   │
│         开头                  中间部分                  结尾         │
│       (最高利用)           （严重衰减区）             （高利用）       │
│                                                                     │
│  结论：开头和结尾的信息最容易被模型关注，中间的信息最容易被忽略          │
└─────────────────────────────────────────────────────────────────────┘
```

**对TMS Agent的直接影响：**

```
❌ 错误的Context排列：
[System Prompt] → [10轮对话历史] → [用户新问题] → [RAG检索的关键运维知识]

  问题：RAG检索到的"设备OTA失败根因分析"排在中间某位置
  → LLM可能忽略它 → 诊断建议基于幻觉而非知识库 → 错误OTA指令

✅ 正确的Context排列：
[System Prompt] → [压缩摘要的历史] → [RAG检索的关键运维知识] → [用户新问题]

  关键知识紧贴用户问题放在末尾 → 利用率最高
  历史压缩成摘要放在中间 → 减少中间区域干扰
```

**面试攻击点答案**：
> "我了解Lost in the Middle效应，所以在P2 Context Engine里，RAG检索到的关键文档片段我会放在Context末尾紧贴User Query，而不是夹在History中间。History超过3轮后我会用小模型压缩成摘要，既控制Token量，又避免重要信息被埋到中间区域被忽略。"

│        │                                                           │
│        │ 优点：保留语义，损失少                                       │

│        │ 版本号、已通过HITL的操作）                                   │
│        │                                                           │
│        │ 优点：信息密度最高                                          │
│        │ 缺点：需要结构化Schema定义什么是"关键事件"                     │
└────────┴──────────────────────────────────────────────────────────┘
```

## 三、Temperature与Top_p：参数路由是工程判断，不是调参玄学

### 3.1 先理解原理，才能做路由

**Temperature（温度）**：
```
控制Softmax输出概率分布的"尖锐度"

Temperature → 0（极低）:     Temperature → 1（正常）:     Temperature → 2（极高）:
  概                           概                           概
  率 │████                     率 │██                       率 │█
     │████                        │████                        │██
     │████                        │████                        │███
     │████                        │████                        │████
     └────                        └────                        └────
      词A 词B                      词A 词B                      词A 词B

  每次都选词A（确定性最高）      随机性适中，有概率选B         高度随机，可能选到低概率词
  适合：诊断/命令/结构化输出     适合：通用对话               几乎不用于生产（幻觉风险高）
```

**Top_p（核采样）**：
```
从概率最高的词集合中采样，直到累积概率达到p

Top_p = 0.1:  只从概率最高的词中选（非常保守）
Top_p = 0.9:  从覆盖90%累积概率的词集合中随机选（较丰富）
Top_p = 1.0:  从所有词中按概率采样（最多样）

通常 Temperature 和 Top_p 配合使用，不要同时调高两个
```

## 四、finish_reason生命线：Agent Runtime的安全阀

### 4.1 四种finish_reason完整解析

这是**最重要的一节**。面试官追问概率极高。

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                         finish_reason四种状态全景分析                             │
├──────────────────┬──────────────────────────────────────────────────────────────┤
│ finish_reason    │ 含义、场景、处理策略                                            │
├──────────────────┼──────────────────────────────────────────────────────────────┤
│ ✅ "stop"        │ 含义：模型正常生成完毕，遇到停止符（EOS Token）                  │
│                  │ 场景：正常的对话回答、工具参数生成完整                            │
│                  │                                                               │
│                  │ ⚠️  不能完全信任！                                             │
│                  │ 即使是stop，也要校验JSON格式完整性                               │
│                  │ 因为模型可能生成语法正确但语义错误的JSON                          │
│                  │                                                               │
│                  │ 处理：正常解析 → 校验Schema → 执行工具调用                       │
├──────────────────┼──────────────────────────────────────────────────────────────┤
│ 💥 "length"      │ 含义：达到max_tokens上限，强制截断                               │
│                  │                                                               │
│                  │ 为什么是生产级事故？（必须能讲清楚这个逻辑链）                    │
│                  │                                                               │
│                  │ 事故链：                                                      │
│                  │  1. 模型正在生成工具调用JSON: {"tool":"create_ota_task",        │
│                  │     "params":{"device_filter":"region=华南 AND os=Android11   │
│                  │     AND fail_rate>0.05","version":"2.1.3","count":            │
│                  │     [截断！！！]                                               │
│                  │  2. Agent代码收到残缺JSON → json.loads() 抛异常                │
│                  │  3. 未处理异常 → Agent崩溃 or 静默失败                         │
│                  │  4. 如果有异常处理但不严谨 → 用默认参数调用OTA工具              │
│                  │  5. 最坏情况：count参数缺失→默认全量升级→推送给所有设备→事故     │
│                  │                                                               │
│                  │ 处理策略（V20的选择）：                                         │
│                  │  → 检测到length → 立即抛TruncationError                       │
│                  │  → 不重试（重试仍然截断，浪费Token）                            │
│                  │  → 触发Fallback：降级到规则引擎 or 请求人工干预（HITL）          │
│                  │  → 记录日志：task_id, truncated_content, reason                │
├──────────────────┼──────────────────────────────────────────────────────────────┤
│ 🚫 "content_    │ 含义：内容被模型的安全过滤器拦截                                  │
│    filter"       │ 场景：请求中包含敏感词/违规内容/过于危险的指令                    │
│                  │                                                               │
│                  │ TMS场景示例：                                                  │
│                  │  用户输入"帮我强制删除所有设备上的系统文件" →                    │
│                  │  模型拒绝生成 → content_filter                                │
│                  │                                                               │
│                  │ 处理策略：                                                     │
│                  │  → 不重试（内容不变，重试必然还是被拦截）                        │
│                  │  → 记录审计日志（安全合规要求）                                  │
│                  │  → 返回用户友好提示："该操作不在允许范围内"                      │
│                  │  → 升级到HITL或安全人员审查                                    │
├──────────────────┼──────────────────────────────────────────────────────────────┤
│ 🔧 "tool_calls" │ 含义：模型决定调用工具，输出tool_calls字段而非text                │
│                  │ 场景：ReAct循环中的Action步骤                                   │
│                  │                                                               │
│                  │ 这是正常状态，但处理时有陷阱：                                   │
│                  │  1. tool_calls可能是数组（并行调用多个工具）                     │
│                  │  2. 参数可能包含注入攻击（需要OPA/权限矩阵校验）                 │
│                  │  3. 工具名可能不存在（需要白名单校验）                           │
│                  │                                                               │
│                  │ 处理策略：                                                     │
│                  │  → 校验工具名在白名单中                                         │
│                  │  → 校验调用者权限（工具权限矩阵）                                │
│                  │  → 高危工具（OTA/脚本）→ 触发HITL                              │
│                  │  → 沙箱执行（E2B or Docker）                                   │
└──────────────────┴──────────────────────────────────────────────────────────────┘

  可重试 = 服务端暂时不可用，等待可恢复，且请求本身没有问题

### 4.3 三道面试追问的标准答案

**Q1：你的Agent调用LLM返回了截断JSON，直接解析崩溃，你怎么在架构层面杜绝？**

```
答案（分三层讲，让面试官看到系统性思维）：

第一层：防御性解析
  - 在LLMResponseParser层，每次解析前先检查 finish_reason
  - 非"stop"立即抛 TruncationError / ContentFilterError（专用异常类）
  - 不依赖 json.loads() 异常来发现问题（那是最后防线，不是第一防线）

第二层：结构化输出约束
  - 要求模型输出JSON时使用 response_format={"type": "json_object"}（支持的模型）
  - 或者用少样本示例 + "输出必须是完整可解析的JSON" 约束Prompt
  - 对关键字段设置 max_tokens 计算：工具参数JSON通常不超过500 Tokens，
    动态计算并给max_tokens留足余量（至少1.5倍buffer）

第三层：Fallback兜底
  - TruncationError → 不重试 → 路由到Fallback规则引擎
  - 规则引擎：基于历史OTA规则，生成保守的操作建议
  - 如果规则引擎也无法处理 → 触发HITL，人工审批
  - 整个过程记录到DLQ，任务不丢失

一句话总结："我的设计是三层防御：感知（finish_reason检查）→ 约束（结构化输出）→ 兜底（Fallback+HITL），任何一层都不是单点依赖。"
```

**Q2：日预算还剩1%，来了一个批量OTA紧急任务，你的断路器会拦截吗？如果拦截了，业务怎么兜底？**

```
答案（先讲机制，再讲业务兜底，体现工程与业务的平衡）：

我的三层预算体系中，全局日预算只剩1%时：
  - 普通任务：直接拒绝，返回错误码BUDGET_EXHAUSTED，任务入DLQ等次日
  - 标记为紧急的任务：走特殊通道

如何处理紧急任务：
  1. 任务元数据中有 priority 字段，紧急OTA标记为 priority=CRITICAL
  2. CRITICAL任务触发"紧急配额"逻辑：
     - 通知运维人员"今日预算即将耗尽，当前有紧急任务等待"
     - 由人工授权"临时扩容预算"或"批准走紧急通道"
  3. 即使批准，每个CRITICAL任务仍有独立的任务级预算上限（如$2/task）
     防止单任务把剩余配额耗光

兜底方案：
  - 如果紧急OTA真的无法走LLM（预算完全耗尽/大模型宕机）
  - Fallback到规则引擎：历史成功的OTA任务规则库
  - 降级策略："先跑最小批次（10台）试点，成功后人工确认再扩量"
  
核心原则："紧急优先级是业务决策，预算控制是工程约束，两者都不能抛弃，
           通过人工审批+分级通道来调和矛盾。"
```

**Q3：你区分了可重试和不可重试错误，如果DeepSeek返回503，你退避多久？如果一直503怎么办？**

```
答案（展示Circuit Breaker和熔断策略）：

单次503处理：
  - 第1次：立即重试（排除偶发网络问题）
  - 第2次：线性退避2s
  - 第3次（最后一次）：退避4s
  - 第3次仍失败 → 触发Circuit Breaker计数器

Circuit Breaker状态机：
  - Closed状态（正常）：失败率 < 50%
  - Open状态（熔断）：连续3次失败 → 进入Open，10-30秒内不再发送请求
    → Open期间：所有请求直接走Fallback规则引擎，不走LLM
  - Half-Open状态（探测）：Open超时后 → 放行1个请求探测
    → 成功：回到Closed  → 失败：重新Open（延长熔断时间）

一直503怎么办（体现分层降级能力）：
  Level 1（秒级）：重试+退避（已描述）
  Level 2（分钟级）：Circuit Breaker触发，请求走Fallback规则引擎
  Level 3（小时级）：自动切换到备用模型（如Qwen作为DeepSeek备选）
  Level 4（天级）：人工介入，修改配置，排查原因

面试追问："为什么不无限重试？"
  → "无限重试会占用线程/协程资源，如果下游持续不可用，
     请求会在重试队列堆积，最终导致内存溢出和级联故障。
     熔断是比重试更高层次的保护——与其一直发无效请求浪费资源，
     不如主动断开，保护自己也给下游恢复时间。"
```

### 5.1 finish_reason守门员（最小实现）

```python
from enum import Enum
from dataclasses import dataclass
from typing import Optional

class FinishReason(Enum):
    STOP = "stop"
    LENGTH = "length"
    CONTENT_FILTER = "content_filter"
    TOOL_CALLS = "tool_calls"

class TruncationError(Exception):
    """LLM输出被截断，不可重试"""
    pass

class ContentFilterError(Exception):
    """内容被安全过滤器拦截，不可重试"""
    pass

@dataclass
class LLMResponse:
    content: str
    finish_reason: str
    usage_tokens: int

def parse_llm_response(raw_response: dict) -> LLMResponse:
    """
    第一道防线：在解析JSON之前先检查finish_reason
    """
    choice = raw_response["choices"][0]
    finish_reason = choice.get("finish_reason")
    
    # 🔴 关键：先检查finish_reason，不要默认假设是stop
    if finish_reason == FinishReason.LENGTH.value:
        raise TruncationError(
            f"LLM输出被截断(max_tokens={raw_response.get('max_tokens')})，"
            f"已消耗Token: {raw_response['usage']['total_tokens']}，"
            f"触发Fallback，不重试。"
        )
    
    if finish_reason == FinishReason.CONTENT_FILTER.value:
        raise ContentFilterError(
            "内容被安全过滤器拦截，记录审计日志，不重试。"
        )
    
    # stop 或 tool_calls 才继续处理
    return LLMResponse(
        content=choice["message"].get("content", ""),
        finish_reason=finish_reason,
        usage_tokens=raw_response["usage"]["total_tokens"]
    )
```

### 5.2 Pre-flight Token预算检查

```python
import tiktoken  # 或使用模型API的counting端点

class TokenBudgetGuard:
    def __init__(self, daily_budget: int, task_budget: int):
        self.daily_remaining = daily_budget
        self.task_budget = task_budget
    
    def preflight_check(
        self, 
        system_prompt: str, 
        history: list, 
        tools_schema: list, 
        user_query: str,
        expected_output_tokens: int = 500
    ) -> bool:
        """
        发送LLM请求之前，估算Token消耗，判断是否超预算
        返回 True=通过，False=拒绝
        """
        # 估算各部分Token
        est_system = self._count_tokens(system_prompt)
        est_history = sum(self._count_tokens(str(h)) for h in history)
        est_tools = self._count_tokens(str(tools_schema))
        est_query = self._count_tokens(user_query)
        
        # Input + 预期Output（Output通常比Input贵，要额外计算）
        total_estimated = (
            est_system + est_history + est_tools + est_query 
            + expected_output_tokens
        )
        
        # 三层检查
        if total_estimated > self.task_budget:
            # 触发降级：压缩History or 换小模型
            return self._trigger_task_level_fallback(total_estimated)
        
        if total_estimated > self.daily_remaining:
            # 触发日级拒绝
            raise BudgetExhaustedError(
                f"日预算不足：需要{total_estimated}，剩余{self.daily_remaining}"
            )
        
        return True
    
    def _count_tokens(self, text: str) -> int:
        # 实际用tiktoken，此处简化估算
        return len(text) // 4  # 粗略：4字符≈1Token
```

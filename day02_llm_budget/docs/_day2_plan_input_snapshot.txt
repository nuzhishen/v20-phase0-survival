学习主题

必须理解的问题
1.为什么finish_reason=length是生产级事故？ —— Agent拿到截断JSON，解析崩溃，后续Tool Calling参数残缺，可能触发高危操作（如OTA指令不完整）
2.Retry分类的刚性边界：429（限流）可指数退避重试，400（参数错误）/401（鉴权）重试无意义，finish_reason=length重试只会再次截断
3.Token预算为什么必须Pre-flight？ —— LLM API按请求计费，事后统计无法挽回已发生的成本
手写笔记要求
- 画出Token消耗分布图：System Prompt(500) + History(800) + User Query(200) + Tools Schema(600) + Completion(300) = 2400 tokens，标注哪些是"Agent开发者容易忽略的隐藏开销"
- 手写Retry分类决策表：
错误码/场景是否重试重试策略理由429 Rate Limit是指数退避，最多3次服务端临时过载500/502/503是线性退避，最多2次服务端瞬时故障400 Bad Request否—参数错误，重试必败401/403否—鉴权问题，重试无意义finish_reason=length否触发Fallback上下文溢出，重试仍截断超时(>30s)是1次，立即重试网络抖动

今日必须形成的一句话技术判断
在生产级 Agent 系统中，LLM响应的finish_reason不能简单信任默认的stop，因为 length截断会导致Agent拿到残缺JSON并触发不可预期的工具调用。我的设计选择是 在LLMResponse解析层强制校验finish_reason，非stop立即抛TruncationError并触发Fallback到规则引擎，同时将该请求标记为不可重试，代价是 损失了部分长文本任务的自动处理能力，收益是 彻底杜绝了因截断导致的Agent运行时崩溃和安全事故。

面试话术沉淀
我今天设计了一个内嵌预算断路器与finish_reason校验的LLM调用层。
它解决的是生产级 Agent 中的成本失控与截断崩溃问题。
如果没有这个机制，会导致一次超长工具Schema调用耗尽日预算，或Agent拿到截断JSON后解析失败并触发错误OTA指令。
我的方案是三层防线：请求前Token预检拦截超限、响应后强制校验finish_reason（非stop立即抛异常并Fallback）、错误分类决策树区分可重试与不可重试场景。
这个方案的权衡是增加了约20ms的调用层延迟，且需要维护模型单价映射表，收益是将LLM调用从不可预测的消费行为转变为可治理、可度量的工程资源，彻底杜绝截断导致的运行时崩溃。
最终它能带来的价值是月LLM成本降低40-60%，且零截断事故。

# 02 上下文窗口：KV Cache / Lost in the Middle / 长对话膨胀

> 目标：理解上下文窗口的物理约束，知道为什么"能塞就塞"是错误策略。

---

## 1. 上下文窗口的物理本质

### 什么是上下文窗口？

Transformer 模型在一次推理中能"看到"的最大 Token 数量。不同模型上限不同：

| 模型 | 上下文窗口 |
|------|-----------|
| GPT-4o | 128K tokens |
| DeepSeek V3 | 64K tokens |
| Qwen-Long | 1M tokens |
| Claude 3.5 Sonnet | 200K tokens |

**常见误解**：窗口越大越好？— 不对，大窗口有代价。

---

## 2. KV Cache：显存的隐形杀手

### Transformer 注意力机制回顾

```
Attention(Q, K, V) = softmax(QK^T / √d_k) × V

Q = 当前位置的 Query
K = 所有位置的 Key（需要缓存）
V = 所有位置的 Value（需要缓存）
```

### KV Cache 是什么？

推理时，对于已处理过的 Token，它们的 K 和 V 矩阵不需要重新计算，可以缓存复用：

```
第1轮：计算 [token1, token2, ..., token500] 的 KV → 缓存
第2轮：新输入 [token501, ..., token700]，直接复用缓存的 KV
       只需计算新 Token 的 KV + 注意力计算
```

### KV Cache 的显存开销

```
显存占用 = 2（K+V）× num_layers × num_heads × head_dim × seq_len × bytes_per_param

以 7B 模型为例（32层，32头，128维，fp16）：
每1K tokens的KV Cache ≈ 256 MB

实际影响：
- 32K tokens → 8 GB 显存
- 128K tokens → 32 GB 显存（整张 A100 的显存！）
```

**面试话术**：
> "KV Cache 让生成更快，但显存是硬约束。当上下文窗口太长，KV Cache 会撑满显存，
> 导致 OOM 或推理变慢。所以大上下文不是免费的——云端 API 按 Token 计费也体现了这个成本。"

## 3. Lost in the Middle：长上下文的注意力陷阱

### 现象

斯坦福研究（2023）发现：当关键信息放在长上下文的**中间位置**时，模型的检索准确率显著下降。

```
上下文位置 vs 准确率：

准确率
  100% |█                                      █
   90% |██                                    ██
   80% |███                                  ███
   70% |████                                ████
   60% |█████████                      ████████
   50% |████████████            ████████████████
        开头                    中间             结尾
        ↑ 高准确率               ↑ 低准确率       ↑ 高准确率
```

### 为什么会这样？

- Transformer 的位置编码使模型对**首尾**位置有更强的注意力偏置
- 中间信息在多层注意力计算后"被稀释"
- 长度越长，中间信息被遗忘越严重

### 对 TMS Agent 的影响

```
错误做法：把关键设备信息塞在 10 轮历史对话的中间
  [系统提示] [第1轮对话] ... [关键设备SN: ABC123, 固件: 3.2.1] ... [第10轮对话] [当前问题]
  → 模型很可能忽略中间的设备SN，导致OTA任务下发错误设备！

正确做法：关键信息固定在 System Prompt 头部 或 当前 User 消息末尾
  [系统提示 + 当前设备上下文] ... [历史摘要] [当前问题 + 关键参数重复强调]
```

**面试话术**：
> "Lost in the Middle 是我们在 TMS Agent 中遇到的实际问题。设备SN、固件版本这类
> 关键执行参数如果放在历史对话中间，模型有概率搞错。我的解决方案是：这类参数
> 始终在 System Prompt 的设备上下文块中维护，不依赖历史对话传递。"

## 4. 长对话膨胀：每轮调用都在变贵

### 膨胀公式

```
设每轮平均 Token 数为 T（User + Assistant 合计）
第 n 轮调用的输入 Token 数：

Input_n = System_Prompt + (n-1) × T_avg + Current_Query
         ≈ 500 + (n-1) × 500 + 150

第1轮：  650 tokens
第5轮：  2650 tokens  （成本是第1轮的 4倍）
第10轮： 5150 tokens  （成本是第1轮的 8倍）
第20轮： 10150 tokens （成本是第1轮的 15倍）
```

### 生产级上下文管理策略

```
策略1：Sliding Window（滑动窗口）
  只保留最近 K 轮完整对话，丢弃更早的
  ✅ 实现简单  ❌ 丢失早期关键信息

策略2：Summarization（摘要压缩）
  每隔 N 轮，将早期对话压缩为摘要注入
  ✅ 保留关键信息  ❌ 压缩本身消耗Token + 延迟

策略3：Structured Memory（结构化记忆）
  关键实体（设备ID、故障码、操作历史）存 Redis
  每轮只注入相关实体，不放完整历史
  ✅ 精准控制  ❌ 需要额外的记忆提取逻辑

TMS Agent 采用：策略3 为主，策略1 为辅
- Redis 存储：当前任务状态、设备上下文、操作历史
- 对话历史：最多保留 5 轮
- 超出 5 轮：摘要化后存 Redis，不再注入对话历史
```

**面试话术**：
> "长对话不处理，每轮成本指数增长。我的方案是结构化记忆 + 滑窗兜底：
> 关键实体存 Redis 按需注入，对话保留最近5轮，超出部分摘要化。
> 这样 20 轮对话的成本控制在第5轮的 2 倍以内，而不是 15 倍。"

## 6. 面试必背结论

1. **KV Cache 让推理快但吃显存**，长上下文 = 高显存占用 = 高 API 成本
2. **Lost in the Middle**：关键参数（设备SN/固件版本）不要放历史中间，要放头/尾
3. **长对话是成本定时炸弹**：第20轮成本是第1轮的15倍，必须主动管理
4. **生产方案**：结构化记忆（Redis）+ 滑动窗口（5轮）+ 摘要化（超出部分）

# 05 面试攻击点：高频追问 + 标准答案框架

> 这是 CTO / 技术面试官最常用的三道追问。
> 每道题必须能脱稿回答，且经得住"那如果...怎么办"的二次追问。

---

## 攻击点一："截断JSON崩溃——你在架构层面怎么杜绝？"

### 完整问题

> "你的 Agent 调用 LLM 返回了截断 JSON，直接解析崩溃，
> 你怎么在架构层面杜绝这个问题？"

### 分层答案框架

**第一层（30秒版）**：

> "我在 LLMProvider 层强制校验 finish_reason。
> 所有 chat_completion 调用，如果 finish_reason 不是 stop 或 tool_calls，
> 立即抛出 TruncationError，不让任何残缺响应流出 Provider 边界。
> 上层 Agent 捕获 TruncationError 后 Fallback 到规则引擎，不重试。"

**第二层（追问：怎么 Fallback？）**：

> "Fallback 到基于历史操作规则的确定性处理器。
> 比如 TMS 设备故障，如果 LLM 截断了，就查设备错误码对应的预定义处置规则表：
> E1001→重启设备，E1002→推送日志采集脚本，E1003→升级到指定版本。
> 这个规则表是运维团队维护的，不依赖 LLM，100% 确定性。
> 任务同时标记为'已Fallback-需人工复核'，进入审核队列。"

**第三层（追问：Fallback 覆盖不了的新故障怎么办？）**：

> "新型故障没有规则 → 直接升级为 HITL 任务。
> 我们的设计原则是：宁可人工介入，也不让不完整的 AI 决策自动执行。
> 在 TMS 场景，误操作 OTA 可能导致大规模设备变砖，
> 保守策略的代价是多一些人工工单，可接受。"

### 关键技术点清单

- [ ] finish_reason 校验在 Provider 层，不在调用方
- [ ] TruncationError 标记为 NonRetryable
- [ ] Fallback 是确定性规则引擎，不是再调一次 LLM
- [ ] 降级后任务标记状态变更，触发人工审核流

## 攻击点二："日预算剩1%，紧急OTA任务来了，断路器拦截了怎么办？"

### 完整问题

> "日预算还剩 1%，来了一个批量 OTA 紧急任务，你的断路器会拦截吗？
> 如果拦截了，业务怎么兜底？"

### 分层答案框架

**第一层（30秒版）**：

> "会拦截。我的设计是断路器优先于业务紧迫性——这是硬约束，不能绕过。
> 但我设计了紧急通道：预算超限时，自动触发紧急通知（Webhook + 短信），
> 让平台管理员决定是否手动扩容当日预算或手动执行操作。
> Agent 系统本身不自动扩容预算，这个权限必须是人的决策。"

**第二层（追问：这不是让业务等很久吗？）**：

> "是的，这是有意为之的权衡。我们有两个设计选项：
> 选项A：Agent 在紧急情况下自动扩容预算 → 成本失控风险，可能日消耗100倍
> 选项B：Agent 上报紧急状态，人类决策扩容 → 最坏情况等待 5-15 分钟
> 在 TMS 场景，设备管理 SLA 是分钟级，不是秒级，5 分钟人工决策可接受。
> 如果是真正的 P0 事故，紧急通道会同步电话通知 on-call，响应更快。"

**第三层（追问：如果没人响应怎么办？）**：

> "没人响应30分钟 → 自动触发降级处理：
> 不调 LLM，直接用规则引擎处理排队中的紧急任务。
> 这个降级路径不消耗 Token 预算，专门为预算耗尽场景设计。
> 规则引擎处理完后，结果标记为'规则降级-需事后复核'，次日人工确认。"

### 关键技术点清单

- [ ] 断路器拦截是硬约束，人工决策扩容
- [ ] 紧急通知链：Webhook → 短信 → 电话 on-call
- [ ] 降级路径（规则引擎）不消耗预算
- [ ] 所有降级结果标记状态，事后可审计

## 攻击点三："DeepSeek 503，你退避多久？一直 503 怎么办？"

### 完整问题

> "你区分了可重试和不可重试错误，如果 DeepSeek 返回 503，
> 你退避多久？如果一直 503 怎么办？"

### 分层答案框架

**第一层（30秒版）**：

> "503 走指数退避加 Jitter：第一次 2 秒，第二次 4 秒，第三次 8 秒，最多重试 3 次。
> 加 Jitter（±10%随机）是为了防止多个 Agent 并发重试打垮服务——惊群效应。
> 3 次重试全部失败后，抛 ProviderUnavailableError，不再重试。"

**第二层（追问：一直503，3次重试后然后呢？）**：

> "触发熔断器（Circuit Breaker）：
> 1. 统计最近 60 秒内 503 次数，超过阈值（比如10次）→ 熔断器打开（Open状态）
> 2. Open 状态下，所有到 DeepSeek 的请求立即失败，不发送实际 HTTP 请求
> 3. 同时触发模型路由切换：DeepSeek 熔断 → 自动路由到 Qwen 或 通义千问
> 4. 30 秒后进入 Half-Open 状态，发送一个探测请求，成功则关闭熔断器"

**第三层（追问：切换到备用模型，提示词兼容吗？）**：

> "这正是我用 OpenAI-compatible 适配层的原因。
> 我的 LLMProvider 接口是统一的，DeepSeek/Qwen/通义都实现了相同接口。
> Prompt 格式（messages 数组）在三家都是兼容的。
> 差异点只有两个：
> 1. 某些 Function Calling 的参数格式略有差异 → 在各自的 Provider 实现里做适配
> 2. 不同模型的能力上限不同 → 切换前评估任务类型，复杂推理任务不切小模型
> 实测 Qwen-72B 和 DeepSeek-V3 在 TMS 诊断场景的准确率差异在 5% 以内，
> 可以接受这个降级。"

**第四层（追问：Qwen 也挂了怎么办？）**：

> "三备用穷尽后，Fallback 到规则引擎，同时触发人工告警。
> 这是我们的最后防线。生产系统不能有单点依赖，但规则引擎是本地的，
> 不依赖任何外部服务，只要进程活着就可以执行。"

### 关键技术点清单

- [ ] 503 → 指数退避（2s, 4s, 8s）+ Jitter
- [ ] Circuit Breaker 三状态：Closed → Open → Half-Open
- [ ] 模型路由切换（Fallback Provider）
- [ ] 最终兜底：规则引擎（本地，不依赖外部）

## 综合话术：30秒完整版

**面试开场话术（背诵版）**：

> 我今天设计了一个内嵌预算断路器与 finish_reason 校验的 LLM 调用层。
>
> 它解决的是生产级 Agent 中的两个核心问题：成本失控和截断崩溃。
>
> 如果没有这个机制，一次超长工具 Schema 调用会耗尽日预算，
> 或者 Agent 拿到截断 JSON 后解析失败并触发错误的 OTA 指令。
>
> 我的方案是三层防线：
> 第一层，请求前 Token 预检，超限直接拒绝，不发 HTTP 请求；
> 第二层，响应后强制校验 finish_reason，非 stop 立即抛异常并 Fallback；
> 第三层，错误分类决策树区分可重试和不可重试，429 指数退避，length 绝不重试。
>
> 这个方案的权衡：增加了约 20ms 的调用层延迟，且需要维护模型单价映射表。
> 收益是：月 LLM 成本降低 40-60%，且零截断事故。

## 备用追问 QA

**Q：为什么不用 LangChain 的回调来做这些？**
> A："LangChain 的回调是观测用的，不是控制流的。我需要在 response 返回时做
> 强制拦截，而不是观测后通知。如果用 LangChain，我还是要在回调里手写这套逻辑，
> 不如直接在 Provider 层实现，依赖更少，测试更容易。"

**Q：预算是内存存的，服务重启后怎么办？**
> A："今天是 MVP 版本，内存存储是有意为之——先跑通逻辑，注释里标注了
> '生产需替换为 Redis'。Redis 版本是 Week 5 的工作，P1 V3 里做。
> 现阶段的重点是验证断路器逻辑正确性，不是做存储持久化。"

**Q：Retry 最大次数 3 次是怎么定的？**
> A："这是个经验值。429 Rate Limit 的退避时间是 2s+4s+8s=14秒，
> 如果14秒后还在限流，说明问题更严重，超过 3 次重试通常不会成功，
> 反而在排队让问题变得更糟。同时 3 次 × 最大8秒 = 最长24秒延迟，
> 对 TMS 任务的用户响应仍在可接受范围内。"
